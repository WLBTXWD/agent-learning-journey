## 感悟
为什么 coding 会成为模型前沿的主战场？
https://www.zhihu.com/question/2038217789668684755/answer/2038355306841887787

METR 在 2025 年 3 月发了一篇 task length doubling 的论文[12] 把这件事的进展量化了：模型能可靠完成的任务时长，每七个月翻一倍。
Claude 3.7 Sonnet 时代能稳定搞定的任务长度大概是人类工程师 50 分钟的量，o3 是 110 分钟。
论文外推这条线，五年后 AI 能独立做一个月长度的软件工程任务。

真正值钱的能力正在从"写代码的速度"快速转向**"判断 AI 输出对不对的能力"**。前者会越来越不稀缺，后者会越来越贵。

可验证奖励训不出"AI 看起来对但其实错"那一档的判断力。
RLVR 让模型在 grep 得通、编译得过、单测能跑的代码上越做越像专家。
但软件工程里真正最贵的那一类 bug，恰好都活在 grep / 编译 / 单测的盲区里：并发死锁、内存生命周期、跨线程数据竞争、API 微妙的语义边界、上线灰度才暴露的环境差异。
这些东西没有验证函数能事先给出 0/1 信号。会写代码的人越来越多，能在 AI 输出里指出"这段虽然测试通过但在生产会炸"的人，没那么多。
这是一个反过来变得更稀缺的能力，而且它正好不是 RLVR 能训练出来的那种能力。
更具体一点说，2026 年的中高级开发者其实在做一件 周恩来总理[44] 时代搞外交的事，你不再亲自写每一份文件，你的工作是判断手下交上来的东西能不能用。
能不能区分一个看起来对但实际有竞态条件的并发代码、能不能识别一个测试通过但其实绕过了真问题的"作弊解"，这件事的价值比五年前的"会用 Auto Layout"高得多。
这个判断不是危言耸听。METR 那条 task length 曲线如果按 7 个月翻一倍跑下去，五年后 AI 能独立做一个月长度的软件工程任务。
到那个时候，程序员的工作就和今天的 PM、TPM 没有本质区别，你给方向、你审查产出、你为后果负责。代码会从你的输出物退到原料，判断才是你真正交付的东西。

## llm服务端kv cache原理：
### 二、这个 KV 缓存到底存了什么？
大模型推理分两步：
Prefill 阶段（处理输入 Prompt）
逐 Token 计算每一层注意力 KV 向量，消耗大量算力，是长文本最耗时、最贵的环节。
Decode 阶段（逐字生成回答）
DeepSeek 会把完整、连续、从第 0 个 Token 开始的输入前缀，计算好的 KV 向量压缩后存在分布式 SSD 硬盘缓存里。
下次请求开头有一模一样的长前缀时：
重复前缀：直接读取缓存好的 KV，不用重新走 Prefill → 命中缓存 token，低价计费
后面新增不一样的文字：重新计算 KV → 未命中缓存 token，原价计费
关键匹配规则（必须全部满足才算命中）
必须从头完全匹配前缀，中间重复、尾部重复都不算；只要前缀有一个 Token 改动，后面全部失效；
最小缓存单元 64Token，小于 64 的短文本不会缓存；
同账号隔离，不同 API Key 缓存互不通用；
缓存是尽力机制，闲置一段时间会自动清理，不永久保存。
### 三、用量两个指标含义（对应控制台统计）
总输入 Token = 缓存命中输入 Token + 缓存未命中输入 Token
输入（命中缓存）prompt_cache_hit_tokens
本次输入里，和历史请求前缀完全重复、直接读取缓存 KV、无需重新计算的文本 Token；
计费单价极低，通常只有未命中的 1/10～1/120。
输入（未命中缓存）prompt_cache_miss_tokens
本次输入中新出现、没有缓存、需要服务器完整算力计算 KV 的文本 Token；标准原价计费。
举个直观例子：
你固定用一段 8000Token 的系统提示词 + 知识库文档，每次只换最后 100 字用户提问：
8000Token 固定前缀 = 缓存命中输入
末尾 100 字新问题 = 缓存未命中输入
账单只按 100 字原价收费，8000 字按极低缓存价计费，延迟大幅下降。
### 四、为什么会出现缓存命中？哪些场景高频命中
1. 固定 System 提示词 + RAG 固定知识库（最常见）
Agent、知识库问答场景，每次请求开头都是完全一样的角色设定、规则、参考文档，只有末尾用户问题变化。
多次调用后，固定长前缀落盘缓存，后续请求直接命中。
2. 多轮连续对话
一轮对话完整消息：系统提示+历史对话1+历史对话2+当前提问
下一轮对话只是在末尾追加新用户消息，前面全部上下文前缀完全一致，大量 Token 命中缓存。
3. 批量处理、代码解释、模板化 Prompt
批量摘要、批量代码审查，共用超长模板，仅少量变量改动；重复前缀反复复用。
4. 开发调试场景
反复发送一模一样的完整 Prompt 调试 Agent/Hooks，整条输入前缀完全匹配，全部命中缓存。
五、极易无法命中缓存的反面场景
System Prompt 内置动态时间、动态随机数，每次前缀都有一字变化；
每条用户提问完全独立、无任何共用长文本；
单次输入总长度小于 64Token，达不到缓存最小单元；
两次请求前缀文字存在任何标点、空格、字符差异；
间隔太久，服务器自动清理了历史缓存。
